# META_LEARNING Framework

> **目标**：把"协作交付"升级为"可迁移、可复用、可运营"的工程能力。

**第一性原理**：
- **能力**不是一次把事做出来，而是可复现的"决策与交付机制"
- **学习**发生在真实任务中：假设 → 实验 → 反馈 → 修正
- **复利**来自把机制固化为资产：模板 / 自动化 / 评估 / 标准

---

## Part 1: Quick Start (3 minutes)

### 你在优化什么？

> **北极星**：用最小的额外成本，把每次任务的有效机制沉淀为可复用资产。

### 能力层级（一眼判别）

```
L1: 能把任务跑出结果（会用工具、能复现）
L2: 能对交付负责（DoD + 验证）
L3: 能做出可复用系统（契约 + 参数化 + 复用3次以上）
L4: 能设计可扩展平台（可治理 + 可观测 + 多人可用）
```

```

**任务后（90秒）**：
```

## Part 2: The Framework (15 minutes)

### 2.1 层级量表（Rubric）

**规则**：只看证据，不看工具。出现反证，默认降级。

#### L1: 工具使用者

**证据**（满足≥3条）：
- 能在项目里跑通：环境/命令/依赖/基本结果可复现
- 能读改AI产出：修编译/类型/小缺陷
- 能提供足够上下文：文件路径、日志、复现步骤

**反证**（出现任一条）：
- 只能"复制粘贴试运气"，无法稳定复现
- 不能解释改动原因，只能反复重试

#### L2: 协作工程师

**证据**（满足≥3条）：
- 任务有清晰 **DoD（验收标准）**
- 有可执行 **验证计划**（测试/检查清单/回归步骤）
- 能做方案权衡：复杂度/风险/维护成本
- 能小步交付：把大任务切片

**反证**（出现任一条）：
- 无DoD；或DoD不可验证
- 不验证就合并；问题靠运气

#### L3: 流程自动化者

**证据**（必须同时满足3条）：
1. **可复用工件**：有清晰I/O契约 + 参数化 + 最小文档
2. **可运营**：有基础日志/错误信息可行动
3. **复用收益真实**：复用≥3次，每次人工介入≤5分钟

**反证**（出现任一条）：
- 每次都要"手工改prompt/改路径"才能成功
- 失败只能重跑，没有可诊断信息
- 维护成本 > 节省时间

#### L4: 系统架构师

**证据**（必须同时满足3条）：
- 多个自动化可组合：接口/契约清晰
- 有治理要素：观测/权限/版本/成本意识
- 可被他人使用：**5分钟上手**

**反证**（出现任一条）：
- 系统只能作者本人使用
- 无版本/兼容策略，升级导致频繁破坏

### 2.3 资产账本（Asset Ledger）

> 自动化不是目标；**净收益为正**的自动化才是资产。

#### 资产分级

**黄金资产**：
- 高频（≥每周1次）、边界清晰、维护低
- 策略：优先投入，做到L3稳定复用

**灰色资产**：
- 可能复用，但边界未清晰/需求变化大
- 策略：先做模板/半自动，等稳定后再系统化

**腐烂资产**：
- 一次性、强耦合、外部依赖多、易过期
- 策略：坚决不系统化，走L2快速交付

### 2.4 Eval Minimum

> L2/L3默认使用；L3自动化必须至少满足1-1-1。

**三分法**：
- **Spec Correctness**：是否满足DoD
- **Behavior Correctness**：边界/异常下是否稳定
- **Regression Safety**：是否破坏既有能力

**L3的最小1-1-1**：
1. 1个happy-path示例（默认配置能跑通）
2. 1个failure-path示例（失败时给出可行动错误）
3. 1个回归断言（快照/黄金文件/最小测试）

---

### 2.5 元技能（Meta Skills）

> 从L2→L3→L4的核心不是"写更多代码"，而是这些元技能更硬。

**约束清单**：每个任务强制写5条约束（数据/性能/安全/边界/依赖）

**失败模式枚举**：每个L3资产至少列5个失败模式 + 信号 + 恢复动作

**基线与对照**：自动化前先记录手工baseline（频率×时间）

**变更控制**：每次重构必须给：回滚策略 + 版本标记 + 兼容说明

## Part 3: Reference (按需查阅)

### 3.1 两条关键纪律

1. **主文档必须可日用**：任何表单超过3分钟就要砍
2. **只在黄金/灰色资产上追求L3+**：腐烂资产直接走L2快交付

### 3.2 个人偏好配置

你的偏好应该存储在 `.claude/settings/learning_preferences.json`：

```json
{
  "core_goal": {
    "primary": "capability提升（80%权重）",
    "secondary": "完成任务是达成提升的方式"
  },
  "current_level": {
    "current": "L2~L3",
    "target": "深入掌握L1~L3",
    "focus": "每次聚焦1-2个可验证的L3行为"
  },
  "high_frequency_context": {
    "task": "学习本repo内容",
    "frequency": "每天1-2小时"
  },
  "collaboration_style": {
    "role": "严厉教练 + 写作伙伴",
    "expectations": [
      "直接指出问题",
      "提供可执行建议"
    ]
  }
}
```

### 3.3 从L2突破到L3的关键

**识别瓶颈**：
- 哪些任务你每周重复做≥3次？
- 这些任务是否有清晰的I/O？
- 是否有边界情况需要手工处理？

**设计自动化**：
1. 定义输入（参数化）
2. 定义输出（契约）
3. 处理边界情况（失败模式）
4. 添加日志/错误信息（可运营）

**验证复用**：
- 复用≥3次
- 每次人工介入≤5分钟
- 净收益 > 开发时间

### 3.4 常见失败模式

| 反证指标 | 说明 | 解决方案 |
|---------|------|---------|
| 每次都要手工改 | 缺少参数化 | 提取变量，配置化 |
| 失败只能重跑 | 缺少诊断信息 | 添加日志，错误分类 |
| 维护 > 节省 | ROI为负 | 降级到L2或废弃 |
| 只能作者用 | 缺少文档/标准化 | 写5分钟上手指南 |
